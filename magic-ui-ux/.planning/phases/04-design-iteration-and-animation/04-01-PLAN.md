  <name>Task 1: Create /iterate command with variant and edit flows</name>
  <files>commands/iterate.md, SKILL.md</files>
  <action>
Create `commands/iterate.md` with the following structure, following the exact frontmatter and formatting conventions established in `commands/design.md`:

**Frontmatter:**
```yaml
description: "Iterate on existing designs -- generate variants or edit screens with change requests"
allowed-tools: Read, Write, Edit, Bash(ls:*,cat:*,mkdir:*), Task, google_stitch_edit_screens, google_stitch_generate_variants
argument-hint: "<page> [--variants | --edit <changes>]"
```

**Pre-Flight Check:** Same pattern as design.md -- check .ui-ux/ exists, read state.json, read tokens.json (REQUIRED), read branding.md. Additional check: verify the specified page has status "designed" in state.json (must have at least one existing screen to iterate on). If page is not designed, halt: "Run /magic-ui-ux:design {page} first."

**Two Modes:**

1. **Variant Mode** (`/iterate homepage --variants` or default when no flag):
   - Read the page's existing screen from state.json (get stitchScreenId)
   - Read the page's UX brief from .ui-ux/briefs/{page}-ux-brief.md for context
   - Call `generate_variants` with the existing screen ID
   - User selects preferred variant or requests more
   - Save selected variant to state.json: new screen entry with incremented variant number, add to page's screenIds array
   - Update page status to "iterated"

2. **Edit Mode** (`/iterate homepage --edit "make the hero background darker, increase CTA button size"`):
   - Read the page's existing screen from state.json (get stitchScreenId)
   - Read tokens.json to validate the edit request doesn't violate design system (e.g., if user says "change to red" but brand has no red, warn but allow with confirmation)
   - Craft an edit prompt that incorporates: the user's change request, relevant token values to maintain consistency, reference to the UX brief's psychology rationale for the affected sections
   - Call `edit_screens` with the screen ID and crafted edit prompt
   - Save the edited screen to state.json: new screen entry (same page, incremented variant), update page's screenIds
   - Update page status to "iterated"

**Screen Selection (for pages with multiple screens):**
If a page has multiple screens in state.json, present a numbered list and let user select which screen to iterate on. Default to the most recent screen (highest variant number).

**Token Constraint Enforcement:**
Before any Stitch call in edit mode, check the edit request against tokens:
- If request contradicts token values (e.g., "change primary color to green" when primary is #1A1A2E), warn: "This change conflicts with your design system. Proceed anyway, or update branding first?"
- Always include token context in edit prompts so Stitch maintains consistency for unchanged elements

**Output:** Show the user: page name, mode used (variant/edit), new screen ID, Stitch project URL, variant number, and suggestion to view in Stitch.

**Error Handling:** Follow same patterns as design.md -- Stitch MCP not connected, generation failure (save context for retry), page not found in state.

Also update `SKILL.md` to add the /iterate command to the Commands table:
```
| `/magic-ui-ux:iterate` | Iterate on existing designs -- generate variants or edit screens |
```

Extend the existing UI Agent (`agents/ui-agent.md`) by adding two new workflow sections AFTER the existing Step 7 (Output Summary). Do NOT modify the existing 7 steps -- append new sections.

**Add "## Iteration Workflow" section with two sub-workflows:**

**Variant Generation Workflow (4 steps):**
1. Pre-Flight: Same as main workflow pre-flight, plus verify target screen exists in state.json
2. Context Loading: Read the original page's UX brief to understand psychology rationale. Read tokens for design system context. Load the target screen's stitchScreenId from state.json.
3. Variant Call: Call `generate_variants` with the stitchScreenId. The variants are generated by Stitch using the original screen as a base -- no additional prompt needed (Stitch handles this internally).
4. Persistence: For each variant returned, add a new screen entry to state.json with incremented variant number. Add new screen IDs to the page's screenIds array. Update page status to "iterated". Update updatedAt timestamp.

**Screen Edit Workflow (5 steps):**
1. Pre-Flight: Same as variant pre-flight
2. Edit Analysis: Parse the user's change request. Cross-reference with tokens.json to identify conflicts (color changes, typography changes, style changes). Flag conflicts for user confirmation.
3. Edit Prompt Crafting: Build an edit prompt that combines: (a) the user's specific change request, (b) token values for elements NOT being changed (to maintain consistency), (c) UX brief psychology notes for affected sections (so edits don't break psychological intent). Apply the same quality checklist from stitch-prompt-guide.md -- at least 3 hex values, font families, UI style adjectives.
4. Edit Call: Call `edit_screens` with the target stitchScreenId and the crafted edit prompt.
5. Persistence: Same pattern as variant persistence -- new screen entry, updated page status, updated timestamp.

**Add Quality Rules for Iteration (append to existing Quality Rules section):**
6. Edit prompts MUST include token values for unchanged elements. If editing the hero, still include correct colors/fonts for the rest of the page.
7. Variant selection MUST be saved to state.json before reporting success. Never report a variant as selected without persisting it.

**Do NOT remove or modify anything in the existing 7-step workflow or existing quality rules 1-5.** Only append.
