```
---
name: persistence

  Manages the .ui-ux/ persistence folder. Handles initialization, reading, and writing
  of design tokens, project state, and screen references. Every command MUST read from
  .ui-ux/ before any design operation.
---
```

Include these sections:

**Purpose**: .ui-ux/ is the single source of truth for all design operations. It persists design tokens (JSON), screen IDs, project state, branding guidelines, and UX briefs across sessions.

**Folder Structure**:
```
.ui-ux/
  tokens.json        # Design system tokens (colors, typography, spacing, patterns)
  state.json         # Project state (screens, pages, Stitch IDs, status)
  branding.md        # Human-readable design system description
  briefs/            # UX briefs per page ({page}-ux-brief.md)
  references/        # Reference screenshots and assets
```

**Initialization** (first use): When `.ui-ux/` does not exist:
1. Create the directory: `mkdir -p .ui-ux/briefs .ui-ux/references`
2. Create `state.json` with projectName (ask user), niche (ask user), createdAt timestamp, empty screens/pages arrays, brandingComplete: false
3. Do NOT create tokens.json yet — that comes from /branding command (Phase 2)

**Reading** (before every design operation):
1. Check `.ui-ux/` exists. If not, run initialization.
2. Read `state.json` for current project status
3. Read `tokens.json` if it exists (branding complete)
4. Read `branding.md` if it exists (human-readable reference)

**Writing** (after design operations):
1. Update `state.json` with new screen IDs, page statuses, timestamps
2. Never overwrite tokens.json during design — only /branding writes tokens
3. Save UX briefs to `briefs/{page}-ux-brief.md`

**Schema References**: Point to `schemas/tokens.json` and `schemas/state.json` for validation.

**Key Rule**: "Always read .ui-ux/ before generating. Never design blind."
